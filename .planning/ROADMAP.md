## 阶段详情

### Phase 1: 移植验证与修复

**目标**: 全面审计 FreeRTOS 版本的移植完整性，确保所有裸机功能正确迁移，架构符合 FreeRTOS 最佳实践。

**需求**: VERIFY-01, VERIFY-02, VERIFY-03, VERIFY-04, VERIFY-05, VERIFY-06, VERIFY-07

**依赖**: 无

**成功标准**:
1. 29 个 `.c` 源文件全部存在且功能入口对齐裸机版本 ✅
2. 所有传感器驱动功能正常（数据格式、时序一致）✅
3. GPRS/MQTT 通信链路建立和收发正常 ✅
4. 任务划分合理，无长时间占用 CPU 的任务 ✅
5. 临界区正确保护（互斥锁覆盖 USART1/3、SPI Flash、sensorBuf）✅
6. ISR 处理符合 RTOS 规范（极简、信号量 defer）✅
7. SPI Flash 读写、环形缓存逻辑正确 ✅
8. 休眠唤醒流程正常，功耗无明显增加 ✅
9. delay.c 调度器运行后使用 vTaskDelay ✅
10. gprs.c AT 超时改为信号量驱动（Task 5.1）— 需在 Phase 2 前完成

---

### Phase 2: 采集/发送周期调度

**目标**: 重构 Power/Sensor/Comm 任务的定时调度机制，支持 1s~12h 独立可配的采集周期和发送周期。当发送周期 < 采集周期时，按发送周期频率发送最新数据。

**需求**: PERIOD-01, PERIOD-02, PERIOD-03, PERIOD-04, PERIOD-05

**依赖**: Phase 1 (确保传感器采集、MQTT 发送基础功能正常; 含 gprs.c AT 超时信号量改造 — Wave 5.1)

**成功标准**:
1. 采集周期 1s~12h 可通过 MQTT 命令配置，步进 1s
2. 发送周期独立配置，不受采集周期限制
3. 发送周期 < 采集周期时：每秒/每发送间隔发送最新已采集数据
4. 部分参数更新场景：缺失参数标记为无效值，有效参数正常上报
5. 发送周期 ≥ 采集周期时：行为与现有逻辑一致（采完再发）
6. RTC 定时精度满足秒级调度需求

---

### Phase 3: 断网缓存与补发

**目标**: 实现断网情况下的数据缓存和重连补发机制。按发送周期缓存原始数据 + 时间戳，联网后按 FIFO 顺序补发，保证数据不丢失。

**需求**: CACHE-01, CACHE-02, CACHE-03, CACHE-04, CACHE-05, CACHE-06

**依赖**: Phase 2 (周期调度机制就绪)

**成功标准**:
1. 断网时自动检测并开始缓存（按发送周期频率）
2. 每条缓存记录包含原始传感器值 + 时间戳
3. 缓存数据按 FIFO 顺序写入 SPI Flash
4. 联网后自动从最早未发送数据开始补发
5. 补发完成后自动切回实时发送模式
6. 缓存满时自动覆盖最旧数据
7. 1s 发送周期场景：模拟断网 5 分钟 → 联网后 300 包数据全部补发成功
8. 平台验证每分钟 60 包数据完整、时序正确

---

### Phase 4: 集成测试与验证

**目标**: 全场景测试覆盖，验证功能正确性、稳定性、数据完整性。

**需求**: VERIFY-01~07, PERIOD-01~05, CACHE-01~06

**依赖**: Phase 1, 2, 3

**成功标准**:
1. 正常采集发送场景：全部传感器数据正确上传
2. 发送周期 < 采集周期场景：15 分钟测试，数据包间隔均匀
3. 断网重连场景：3 次断网/重连循环，数据零丢失
4. 混合周期场景：变更配置 3 次，每次正确响应
5. 长期稳定性：24 小时运行无死机、无内存泄漏
6. 功耗评估：与裸机版本对比无明显增加
7. 测试报告输出
