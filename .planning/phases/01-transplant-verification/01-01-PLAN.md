## Wave 5: 审查驱动的修复 (来自 REVIEWS.md)

### Task 5.1: gprs.c 信号量超时改造
**问题**: 原计划将其 defer 到 Phase 2，但 Phase 2 是周期调度而非 AT 引擎重构。gprs.c 的 18 个 AT 命令 `for/count--` 超时循环会独占 Comm 任务，使任务优先级失去意义。

**范围**: 不重构整个 AT 引擎为事件驱动（太大），而是将 AT 超时循环从忙等待改为信号量等待，这是最小侵入式修复。

<read_first>
- Project/src/gprs.c: 搜索 `for` 超时模式（行 ~3500-5000 区域，AT 命令发送/接收）
- `extern SemaphoreHandle_t xUSART1Mutex`
</read_first>

<acceptance_criteria>
- gprs.c 中每个 `for(count=0;count<SOME_LIMIT;count++)` AT 超时循环替换为 `xSemaphoreTake(xATTimeoutSemaphore, pdMS_TO_TICKS(timeout_ms))`
- 添加 `xATTimeoutSemaphore` 在 main.c 中创建
- 构建零警告
- Comm 任务中不再有毫秒级忙等待
- 已知局限：仅改造超时机制——整体 AT 命令状态机维持裸机模式
</acceptance_criteria>

<action>
- 创建 `xATTimeoutSemaphore` (计数信号量，初始 0) 在 main.c
- 每个 `for(count=0;...) { delay_1ms; check_flag }` 替换为:
  ```c
  if(xSemaphoreTake(xATTimeoutSemaphore, pdMS_TO_TICKS(timeout)) == pdTRUE) { /* success */ }
  ```
- 在 UART RX ISR 或数据到达路径中调用 `xSemaphoreGive(xATTimeoutSemaphore)`
- 保留轮询作为 fallback (信号量超时后检查一次标志)
</action>
